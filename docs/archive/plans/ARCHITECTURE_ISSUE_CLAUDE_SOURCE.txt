 Architecture Review — TheFortressSimulation     
                                                  
  Scope. I read project files, the solution, and ~30 source files directly (FortressState, GameStateManager, TickScheduler, both ContentRegistry classes,
  SimulationDiffApplicator, ItemManager, World, Program, TestRunner) and used parallel sub-agents to scan the larger subsystems (FortressState, App/Jobs, Navigation, UI/Input,
   determinism/diff). I personally verified every claim labelled VERIFIED below with grep/read against the source. Claims labelled (via sub-agent) are sub-agent findings I
  have not re-checked line-by-line but the surrounding evidence corroborates.
                                                                                                                                                                             
  Verdict in one paragraph. The repo's design documents (CONCURRENCY_MODEL, UPDATE_ORDER, DIFF_LOG_AND_MERGE_STRATEGIES, DETERMINISM_CI, etc.) describe a credible             
  architecture: layered assemblies, a deterministic tick loop, RNG streams, a DiffLog write-barrier with PostTick applicator, an EventBus, a CommandQueue, content registries  
  with mod stack. The implementation honours only some of that. Several pillars exist as instantiated objects with no real producers, several others are wired up but bypassed 
  by inline mutation, and the layering has decayed into a god-class App layer that mutates the simulation directly. The codebase is alpha/prototype-quality wearing the clothes
   of a deterministic ECS-style engine, and shipping it as the foundation for a "large, maintainable" colony sim will compound the existing debt unless the seams are repaired 
  now.

  1. Project layering and dependencies

  Project graph (VERIFIED from csproj files).

  Core ─────────────────────────────────┐
    ▲          ▲                        │
    │          │                        ▼
  Simulation ──┤             (Newtonsoft.Json 13.0.3)
    ▲          │
    │          │
    ├──────── Navigation       (refs Core + Simulation)
    ├──────── WorldGen         (refs Core + Simulation, SadRogue.Primitives 1.6.0)
    │
    └─ App (WinExe, SadConsole/MonoGame) → refs Core, Simulation, WorldGen, Navigation

  Findings (VERIFIED):

  - Inverted layering. HumanFortress.Navigation/HumanFortress.Navigation.csproj references HumanFortress.Simulation. Navigation is the lower-level concern; it should consume
  an abstract IWorldNavigationView (which exists at src/HumanFortress.Navigation/IWorldNavigationView.cs) and Simulation should implement it. Instead, NavigationManager.cs
  directly types fields as HumanFortress.Simulation.World.World. The abstraction is half-built and ignored.
  - Two empty assemblies on disk. src/HumanFortress.Content/ and src/HumanFortress.Data/ contain no .cs files and no .csproj. They are not in either solution. Either remove
  them or finish them.
  - Two solutions. HumanFortress.sln and TheFortressSimulation.sln both exist. Both reference the same projects. One should be deleted (Directory.Build.props is the
  cross-cutting hook anyway).
  - Inconsistent project hygiene.
    - Core.csproj: TreatWarningsAsErrors=true, AnalysisLevel=latest-all, EnforceCodeStyleInBuild=true.
    - Simulation.csproj: TreatWarningsAsErrors=false, AllowUnsafeBlocks=true, NoWarn=CS1591;CA2231;CA2208.
    - App.csproj: warnings-as-errors off; no analyzers.
    - Directory.Build.props:11 ships a long blanket NoWarn list including CA1031 (catch all) and CA5394 (insecure Random) — the project's claimed determinism is at odds with
  CA5394 being suppressed globally.
  - Package version split. Simulation pins TheSadRogue.Primitives 1.6.0-rc1 while WorldGen pins 1.6.0 (release). Allowed by NuGet's nearest-wins, but a smell.
  - InternalsVisibleTo to a phantom test assembly. Core.csproj declares <InternalsVisibleTo Include="HumanFortress.Core.Tests" /> but no test project exists anywhere in the
  repo (no folder, no .csproj). The "tests" are PhaseTests.cs and TestRunner.cs inside the App WinExe project — see §9.
  - Heavy library at the bottom of the stack. Core takes a hard dependency on Newtonsoft.Json 13.0.3. This is fine, but Core/Content/ContentRegistry.cs:7 also has using
  Newtonsoft.Json.Schema; (a separate, unreferenced package) that is never used in the file (JSchema|Validate greps yield nothing). It's a dead using that would break the
  build if any analyzer were strict.

  Recommendation. Invert Navigation→Simulation; collapse the two solutions; delete the empty assemblies; align package versions and warning policy across projects; either add
  a real test project or remove the InternalsVisibleTo and move Phase/TestRunner out of App.

  2. Two ContentRegistry classes (one is dead and silently empty)

  VERIFIED. Two distinct types share the name ContentRegistry:

  ┌──────────────────────────────────────────┬─────────────────────────────────────┬─────┬──────────────────────────────────────────────────────────────────────────────────┐
  │                   Path                   │              Namespace              │ LOC │                                      Status                                      │
  ├──────────────────────────────────────────┼─────────────────────────────────────┼─────┼──────────────────────────────────────────────────────────────────────────────────┤
  │ Core/Content/ContentRegistry.cs          │ HumanFortress.Core.Content          │ 585 │ Live, loaded at Program.cs:61                                                    │
  ├──────────────────────────────────────────┼─────────────────────────────────────┼─────┼──────────────────────────────────────────────────────────────────────────────────┤
  │ Core/Content/Registry/ContentRegistry.cs │ HumanFortress.Core.Content.Registry │ 853 │ Singleton accessed at GameStateManager.cs:252, but LoadContentAsync is never     │
  │                                          │                                     │     │ called from production                                                           │
  └──────────────────────────────────────────┴─────────────────────────────────────┴─────┴──────────────────────────────────────────────────────────────────────────────────┘

  Two different singletons. Different JSON stacks (Newtonsoft vs System.Text.Json). Different ID schemes (alphabetical sort → stable ushort handles vs. monotonic
  auto-increment from 10).

  The kicker: ItemManager.SetDependencies(world, HumanFortress.Core.Content.Registry.ContentRegistry.Instance) at GameStateManager.cs:252 injects the dead registry into
  ItemManager. Because no one ever calls LoadContentAsync on it, that injected registry is always empty. ItemManager.cs:436 then guards its use with if (_contentRegistry !=
  null && ...) and silently falls back. This is invisible breakage: data-driven item behaviour that would require material properties from the "modern" registry is silently
  disabled.

  This is the most serious single piece of architectural debt in the repository. It also strongly suggests a refactor was abandoned mid-way.

  Recommendation. Pick one. The Registry/ version has the better design (sub-registries, schema validation, mod fallback chain) but isn't wired. Either finish the migration
  (call LoadContentAsync from Program.cs, port the rest of the consumers) or delete Core/Content/Registry/ entirely and grow the old ContentRegistry to cover the missing
  concerns.

  3. The FortressState god-class (VERIFIED via direct read and sub-agent)

  src/HumanFortress.App/States/FortressState.cs — 3,483 LOC, a single class inheriting SadConsole.ScreenObject. It owns ~18 fields including _world (the entire simulation),
  _navManager, _stockpileManager, six UI controllers (UiStore, StockpileUI, OrdersUI, ZonesUI, BuildUI, StockpileQuickUI), rendering surfaces, the snapshot builder, the
  navigation overlay, and the selection tool. It accesses GameStateManager.Instance, GameHost.Instance, WorldMapState.CurrentWorld, ConstructionRegistry.Instance,
  InputBindingsService.Instance, OrdersRegistryService.Instance, and a static lambda registered on ConstructionJobSystem.UiNotifyWorkshopComplete.

  Categories of work inside this one file:
  1. Lazy initialization & fortress generation (lines 172–476)
  2. Keyboard input (1522–1935)
  3. Mouse input (1968–2240)
  4. Order/command submission (2700–2950)
  5. Map render (1168–1285)
  6. UI render (480–651)
  7. Snapshot building (1473–1520)
  8. Camera and tile-info panel state (787–862, 2462–2491)

  Inconsistent mutation paths. All non-mining commands flow correctly through GameStateManager.Instance.EnqueueCommand(cmd) (lines 2735, 2873, 2904, 2929, 2947) — i.e.,
  through the ICommand pipeline. Mining bypasses the queue and writes the world directly:

  // FortressState.cs:2804
  _world.Orders.EnqueueMiningAdvanced(rect, zMin, zMax, simAction,
      priority: 50, createdTick: GameStateManager.Instance.TickScheduler.CurrentTick);

  Same pattern at lines 422 and 435 in the auto-dig path. This is an unprincipled exception (probably "interactive latency"), and it shows the command pipeline isn't trusted
  by its own authors.

  Render and update are entangled. Update(TimeSpan) calls DrawUI() directly (line 169); ProcessKeyboard() and ProcessMouse() also call BuildSnapshot() and DrawUI() (e.g.,
  1607–1609, 1833–1834, 2050). There is no separate render phase — every input event re-renders, and the snapshot is rebuilt synchronously from input handlers.

  Recommendation. Split FortressState into roughly: a FortressScreen (SadConsole glue, lifecycle, dispatch), a FortressInputController (keyboard+mouse → commands), a
  FortressRenderer (snapshot consumer), and a FortressUiCoordinator (drawer/quick-menu/dialog modal state). Drive the mining order through the same EnqueueCommand pipeline as
  everything else so there is one source of truth for replay/audit.

  4. Job systems live in the wrong assembly

  VERIFIED. The bulk of the simulation's job logic lives in src/HumanFortress.App/Jobs/, not src/HumanFortress.Simulation/Jobs/. Sizes from wc -l:

  ├─────────────────────────────────┼───────┼──────────────┤
  │ MiningJobSystem.cs              │ 1,041 │ App (WinExe) │
  ├─────────────────────────────────┼───────┼──────────────┤
  │ TransportJobSystem.cs           │ 824   │ App          │
  ├─────────────────────────────────┼───────┼──────────────┤
  │ ConstructionJobSystem.cs        │ 526   │ App          │
  ├─────────────────────────────────┼───────┼──────────────┤
  │ CraftJobSystem.cs               │ 355   │ App          │
  ├─────────────────────────────────┼───────┼──────────────┤
  │ UnifiedJobsOrchestrator.cs      │ ~185  │ App          │
  ├─────────────────────────────────┼───────┼──────────────┤
  │ ReservationManager.cs           │ ~140  │ Simulation   │
  ├─────────────────────────────────┼───────┼──────────────┤
  │ TransportRequestQueue.cs        │ ~150  │ Simulation   │
  ├─────────────────────────────────┼───────┼──────────────┤
  │ ConstructionMaterialsPlanner.cs │ —     │ Simulation   │
  └─────────────────────────────────┴───────┴──────────────┘

  The App-side job systems are registered into TickScheduler from GameStateManager.StartSimulation (GameStateManager.cs:813–815) and run on the simulation tick thread. They
  take dependencies on World, NavigationManager, DiffLog, ItemsDiffLog, ProfessionAssignments and SchedulerTunings — none of which require SadConsole or MonoGame. There is no
  architectural reason these files must live in the WinExe.

  The likely justification ("they need IPathService / MovementExecutor") is wrong: those live in HumanFortress.Navigation, which the App already references and Simulation
  could reference too (modulo the inverted dependency from §1).

  Consequences. Headless simulation (server, replay, CI determinism harness, mod testing, AI training) is impossible without dragging in SadConsole/MonoGame and a WinExe. The
  "Simulation" assembly is reduced to data-holders and a few planners; the actual game lives in the App.

  Recommendation. Move all App/Jobs/*JobSystem.cs and the UnifiedJobsOrchestrator into a new HumanFortress.Simulation.Jobs.Executors namespace (or a sibling HumanFortress.Jobs
   assembly). Once Navigation is inverted, this is a near-mechanical move. After the move, the Simulation assembly should be runnable from a headless console host that takes
  deterministic save inputs and produces deterministic outputs.

  5. Determinism and the diff/event/command pipeline — mostly aspirational

  This is the most important finding in the review and I verified each item below directly.

  5.1 EventBus is dead (VERIFIED)

  grep for \.Publish\s*\( across src/ returns zero hits. The EventBus is instantiated at GameStateManager.cs:59, exposed via ISimulationContext.EventBus
  (GameStateManager.cs:1027), and given to TestSimulationContext (TestRunner.cs:240). Nothing in the codebase ever publishes an event. Subscribe/Unsubscribe are unused too.
  This is dead infrastructure — and even if it were used, EventBus.Publish iterates a ConcurrentDictionary of handlers in insertion order, which is non-deterministic.

  5.2 RNG streams are dead; non-deterministic Random is live (VERIFIED)

  RngStreamManager is constructed at GameStateManager.cs:60 and then never asked for a stream. Job systems instead do:

  // MiningJobSystem.cs:706, 767
  var rng = new System.Random((int)seed);

  Even worse, WorldGenState.cs:362, 364:

  _params.Width = sizes[new Random().Next(sizes.Length)];        // unseeded
  _params.Difficulty = (DifficultyPreset)new Random().Next(4);   // unseeded

  new Random() is wall-clock-seeded on .NET 8. Determinism of world generation is broken at the very first parameter. The carefully written xoshiro128++ DeterministicRng is
  real and correct, but only FortressGenerator and a couple of WorldGen stages actually use it.

  5.3 DiffLog is wired but treated as audit only (VERIFIED)

  GameStateManager.cs:818 registers OnPostTickApplyDiffs on TickScheduler.PostTick. That handler (lines 1061–1086) does run: it calls _diffLog.MergeAndSort(), hands the result
   to SimulationDiffApplicator.ApplyAll, then applies items diffs, then rebuilds navigation for dirty chunks.

  But the in-source comment at GameStateManager.cs:817 is decisive:

  // Apply diffs after write phase (minimal: currently only used for auditing;
  // runtime updates happen inline)

  And the applicator confirms it (SimulationDiffApplicator.cs:46): default: // Ignore other ops here (Stockpile uses its own applicator).

  In other words: job systems mutate world state directly during WriteTick and emit diffs as a parallel log. Re-applying the diffs is redundant for any op the applicator
  understands; for any op it doesn't (most of them — fluids, fields, placeables, stockpile, etc.), the runtime state was only updated inline and the diff is dropped.

  This breaks the central promise of a diff-log architecture:
  - You cannot replay a save from diffs — the live state is not derivable from the diff stream.
  - CraftJobSystem doesn't even emit diffs for item mutations (verified: _world.Items.RemoveInstance(...) and _world.Items.SpawnItem(...) directly, no diff emission for those
  ops).
  - The 5-op enum that the applicator handles is a fraction of what the world cares about.

  5.4 CommandQueue is live but races with the tick thread (VERIFIED)

  The earlier sub-agent claimed CommandQueue was unused; that is wrong. GameStateManager.cs:723:

  public void Update(double deltaTime) {
      _currentState?.Update(deltaTime);
      if (_tickScheduler.IsRunning && _simContext != null) {
          _commandQueue.ExecuteCommands(_tickScheduler.CurrentTick, _simContext);
      }
  }

  This is called on the main render thread (SadConsole frame pump). Meanwhile TickScheduler.Start() (TickScheduler.cs:139–144) spawns a separate background thread:

  _tickThread = new Thread(TickLoop) { Name = "SimulationTick", IsBackground = true };
  _tickThread.Start();

  So we have:
  - Render thread: drains CommandQueue → ICommand.Execute → mutates world.Orders, world.Zones, etc.
  - Sim thread: runs ReadPhase (parallel.foreach across systems) → barrier → WritePhase → PostTick → applies diffs and rebuilds nav.

  The only synchronization is per-manager (ItemManager._instanceLock for example, or Chunk._writeLock). There is no read/write barrier coordinating render-thread commands with
   the sim thread's read or write phase. A user click that enqueues a CreateZoneCommand can land on world.Zones mid-way through a sim tick that is iterating zones. Today this
  is probably masked by per-manager locks, fast hardware, and a low command rate, but it is a textbook future-bug.

  Worse, this race is invisible to the DiffLog audit: render-thread mutations don't pass through any of the diff infrastructure.

  5.5 TickScheduler is genuinely deterministic in isolation (VERIFIED)

  TickScheduler.ExecuteTick is correct: PreTick → parallel ReadPhase → barrier → serial WritePhase → PostTick. Systems are stable-sorted on Priority once at registration (line
   127). ExecuteSingleTick() exists for tests. If all systems behaved (i.e., used named RNG streams, only mutated via diffs, only read in read phase), the engine would be
  deterministic. None of those preconditions hold.

  5.6 Verdict on determinism

  ~30% live, ~70% aspirational. The infrastructure is well-designed on paper:
  - DeterministicRng (xoshiro128++) — correct, but barely used.
  - DiffLog with merge/sort and system precedence — correct, but bypassed.
  - TickScheduler with fixed step and read/write phases — correct, but its guarantees are forfeited by inline mutation and render-thread commands.
  - EventBus, RngStreamManager — instantiated but never used.

  Replay, rollback, multiplayer, and CI determinism harnesses are not achievable with this codebase as it stands today.

  Recommendation (in priority order).
  1. Make CommandQueue.ExecuteCommands run on the sim thread, between BarrierReached and the write phase, not from the render thread. The render thread should only Enqueue.
  2. Forbid new Random in Simulation, Navigation, App/Jobs, and WorldGen; replace every call with RngStreamManager.GetStream(name).Next(...). Add an analyzer or grep-based CI
  check.
  3. Pick one diff strategy and enforce it: either (a) systems write only diffs in WriteTick and the applicator owns all mutation, or (b) drop the diff log and admit you have
  inline mutation. The current hybrid is the worst of both.
  4. Either light up EventBus (and make its handler iteration order deterministic) or delete it.
  5. Add a headless determinism CI test: spin up World, register systems, run N ticks against a fixed seed, hash the final state. The whole point of the architecture is to
  make this trivial.

  6. World, chunk, and tile model (via sub-agent, spot-checked)

  Chunk is AoS (one TileBase[1024] per chunk, row-major). TileBase is a readonly struct of 7 fields totalling ~10–12 bytes; ~12 KB per chunk. Each Z-level is a separate chunk
  keyed by ChunkKey(ChunkX, ChunkY, Z) (Chunk.cs:270+). World owns chunks in a ConcurrentDictionary<ChunkKey, Chunk> (World.cs:15, 36).

  Observations:
  - Despite design docs talking about ECS/SoA, the implementation is plain managed AoS with per-chunk object _writeLock. No Span<T>, no ArrayPool, no unsafe code despite
  AllowUnsafeBlocks=true.
  - Side-channel data (ChunkPlaceableData, ChunkStockpileData, ChunkZoneData) lives on the chunk via lazy nullable fields and their own internal locks. This is fine but means
  "loading a chunk" actually means creating five potentially separate objects with five locks.
  - LOD policy exists in code (ChunkLifecycleManager.cs 5-level LOD with hysteresis, heat overrides, pin reasons) but disk I/O and memory reclamation are stubbed — chunks
  never actually unload. So today the LOD policy only gates which chunks tick, not memory.
  - World exposes managers as public properties (Service Locator). Combined with GameStateManager.Instance and other singletons, the dependency graph is mostly implicit and
  global.

  7. Navigation (via sub-agent)

  NavigationManager, PathService, PathCache, and DeterministicAStar are the cleanest subsystem in the repo:
  - Path cache keys embed each touched chunk's ConnectivityVersion, so cache invalidation flows naturally from Chunk.SetTile bumping the version.
  - DeterministicAStar.BinaryHeap uses a 4-tier tie-breaker (F → H → G → row-major LocalIndex) — actually deterministic.
  - Multi-Z navigation (stairs/ramps) is supported via per-tile UpRampMask and explicit vertical neighbour expansion.
  - MovementExecutor returns a MovementUpdate struct rather than mutating creature positions, which is the right pattern.

  Weaknesses:
  - Tile-level only (no jump-points, no hierarchical sector graph). For very large maps, A* over thousands of tiles per agent will become a bottleneck.
  - Path queries are synchronous; PathService time-budgets and queues overflow but does not run on a worker.
  - Navigation depends on Simulation.World (see §1).

  This is the subsystem closest to "ready to scale"; it just needs the inversion fix and eventually hierarchical pathing.

  8. UI / input / rendering coupling (via sub-agent)

  - UiRenderer.cs (2,095 LOC) is a static collection of draw routines — actually the cleanest layer; reads-only, takes refs as parameters.
  - UiStore.cs is a mutable state bag (not Redux); UIStateManager is a thin wrapper that exposes Store directly, so the abstraction is leaky.
  - InputHandlerComponent is mostly correct (input → UI state + command enqueue) but calls GameStateManager.Instance.SetProfessionWeight(...) directly (lines 499, 590),
  bypassing the command pipeline.
  - The 9 files under App/Commands/ are real ICommand implementations and flow through GameStateManager.EnqueueCommand → CommandQueue → ExecuteCommands(SimulationContext). The
   pattern is correct; the problem is that the direct mutation paths (mining at FortressState.cs:2804, profession weight in InputHandlerComponent, …) coexist with it.
  - The single-selection model is unified (Selection3D, ISelectionTool) — a good piece of design.
  - Rendering uses a hybrid: RenderSnapshot is built for some overlays, but UI panels poll live world state directly (e.g., UiRenderer.DrawDrawer(..., world) reads
  world.Zones/world.Items from the render thread, while a tick may be writing them).

  9. Tests and CI signals (VERIFIED)

  - No test project exists. No *.Tests.csproj, no *.Test* folder.
  - PhaseTests.cs (888 LOC) and TestRunner.cs are compiled into the App WinExe and run when started with --test or --validate (Program.cs:23–34).
  - This is incompatible with the InternalsVisibleTo "HumanFortress.Core.Tests" declaration in Core.csproj.
  - No xunit / nunit / mstest package is referenced anywhere.
  - No DETERMINISM_CI automation exists in code despite the doc of the same name in the root.
  - The repo contains build/log artefacts at the root (build_output.txt, publish_log.txt, _last_debug_tail.txt, tmp_*.txt x ~30, tmp.cs, $path.bak, nul) — .gitignore is not
  catching them.

  Recommendation. Create a real tests/HumanFortress.Core.Tests project using xUnit. Port the PhaseTests/TestRunner content into proper [Fact]s. Add a dotnet test CI step.
  Clean the root.

  10. Other smells worth naming

  - tmp.cs at repo root is compiled because it sits at the working dir but isn't part of any project; nothing actually compiles it. Still, leaving 25 KB of "scratch" C# at the
   root is risky.
  - Python migration scripts (convert_workshops_and_recipes.py, merge_stoneworks.py, …) live next to the C# source. They belong in tools/.
  - 100+ design docs at the root. The repo's design corpus is larger than its source. There is no INDEX/cross-link discipline; many docs duplicate (UI_REFACTOR_PLAN,
  UI_REFACTOR_PROGRESS, UI_REFACTOR_SUMMARY_CN, UI_ARCHITECTURE_ANALYSIS, QUICK_START_UI_REFACTOR …). New contributors will not be able to tell which docs reflect reality.
  - docs/planning/OPTIMIZATION_SUGGESTION.md and docs/archive/plans/OPTIMIZATION_SUGGESTION_SOURCE_2025.md suggest a previous review already flagged most of this; addressing those before adding more features
  should be the project's priority.

  Top 10 actionable changes (no code edits made)

  1. Fix the dual ContentRegistry. Pick one. The "modern" one is referenced from GameStateManager:252 but never loaded, which silently disables material-driven item behaviour.
  2. Force all sim mutation through CommandQueue + DiffLog, or admit the architecture is inline-mutation and delete the diff log. The hybrid is harmful.
  3. Move CommandQueue.ExecuteCommands to the sim thread, between barrier and write phase. Render thread only enqueues. Removes the render/sim race entirely.
  4. Replace every new Random / new System.Random in Simulation/Navigation/App.Jobs/WorldGen with RngStreamManager.GetStream(name). Lock the CA5394 analyzer back on; add a
  grep-based CI guard.
  5. *Move App/Jobs/JobSystem.cs into Simulation. They have no SadConsole/MonoGame coupling. This unlocks a headless host.
  6. Invert Navigation→Simulation. Simulation should implement IWorldNavigationView, not the other way round.
  7. Split FortressState (3,483 LOC) into Screen / Input / Renderer / UI-Coordinator. Route mining orders through EnqueueCommand like everything else.
  8. Create a real xUnit test project under tests/. Port PhaseTests/TestRunner. Add a determinism harness (N ticks → hash).
  9. Either light up or delete EventBus and RngStreamManager — dead infrastructure rots into "load-bearing zombies" if left around.
  10. Repo hygiene: delete one of the two solutions, delete the empty Content/Data assemblies, delete tmp_*.txt, tmp.cs, $path.bak, nul; move Python tools to tools/; align
  package versions; align TreatWarningsAsErrors.
